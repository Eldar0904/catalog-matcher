1. ТЕКУЩЕЕ СОСТОЯНИЕ (v1)
================================================================================

Назначение:
  Сопоставление внутреннего списка товаров (Our_Items.xlsx) с государственным
  или поставщическим каталогом (Government_Catalog.xlsx).

Стек:
  - Backend: Python, FastAPI, SQLAlchemy, pandas, scikit-learn, rapidfuzz
  - Frontend: React 18, Vite (русский UI)
  - БД: SQLite по умолчанию; PostgreSQL + pgvector в Docker
  - Инфра: docker compose (db, backend, frontend)

Конвейер сопоставления (сейчас):
  Internal Item
    → TfidfRetriever (топ-20 по cosine similarity)
    → DeterministicFilter (boost по коду, порог min_score=0.15)
    → HeuristicRanker (топ-3)
    → MatchResult в БД

Параметры по умолчанию (config.py):
  - top_k_candidates = 20
  - top_n_results = 3
  - min_similarity_score = 0.15

Нормализация при импорте:
  - Каталог normalized_text: name + brand + model + description + technical_specs
  - Позиции normalized_text: item_name + description (код отдельно)
  - Коды: clean_code (uppercase, trim)

Ограничения v1 (намеренные):
  - Нет семантических эмбеддингов и LLM
  - Нет авторизации
  - Слабо на перефразировках и синонимах

================================================================================
2. АЛГОРИТМ СХОДСТВА (ДЕТАЛЬНО)
================================================================================

2.1 TF-IDF + косинусное сходство
  - TfidfVectorizer(ngram_range=(1, 2), min_df=1)
  - Матрица строится по всему каталогу, кэшируется на время запуска
  - Cosine similarity 0..1 между вектором позиции и каждым товаром
  - Берётся топ-20, отбрасываются score = 0
  - Лексическое сходство, не семантика

2.2 Deterministic Filter
  - Точный код (item_code == product.code) → score min 0.95
  - Похожий код (rapidfuzz ratio > 80%) → +0.2 к score, cap 0.9
  - Отсечение score < 0.15

2.3 Heuristic Ranker
  - Сортировка по score, топ-3
  - Лучший автоматически is_selected = 1

2.4 UI интерпретация
  - ≥ 60% — высокий (зелёный)
  - 30–59% — средний (жёлтый)
  - < 30% — низкий (красный)
  - «Подобрано» в статистике: выбранное совпадение ≥ 80%

3. ИЗВЕСТНЫЕ ПРОБЛЕМЫ И ПРОПУСКИ
================================================================================

Архитектурные пробелы (не случайные баги):
  1. Код проверяется ПОСЛЕ TF-IDF — только среди топ-20 кандидатов.
     Если правильный товар на 50-м месте по тексту — код никогда не проверится.
  2. Асимметрия полей: каталог 5 полей, позиции 2 — длинные specs размывают вектор.
  3. top_k = 20 — товар на 21+ месте теряется.
  4. TF-IDF не ловит синонимы и перефразировки («кабель ВВГ» vs «провод силовой»).
  5. Нет fallback при пустом normalized_text, даже если есть код.

Реальные баги:
  1. clean_code(12345.0) → "12345.0", clean_code("12345") → "12345"
     — точное совпадение кодов ломается при числовом формате в Excel.
  2. Нормализация: "3x2.5" → "3x2 5" — разные форматы маркировки снижают сходство.

Отсутствие тестов:
  - Тесты только для normalize.py
  - Нет тестов matching pipeline и API

Соответствие требованиям v1:
  | Требование                    | Статус                          |
  |-------------------------------|---------------------------------|
  | TF-IDF + правила, без LLM     | Да                              |
  | Топ-20 → фильтр → топ-3       | Да                              |
  | Boost за совпадение кода      | Частично (только в топ-20)      |
  | Точные/почти точные названия  | Да                              |
  | Перефразировки                | Ожидаемо слабо                  |
  | Не пропускать по коду         | Нет — может пропускать          |

4. МАСШТАБ 5000 × 3000 — ОЦЕНКА ПРОИЗВОДИТЕЛЬНОСТИ
================================================================================

Не нужен перебор 15 млн пар. Нужен индексированный поиск: для каждой из ~3000
позиций быстро находить топ-K из ~5000.

| Подход                          | Индекс      | 3000 запросов | Итого      |
|---------------------------------|-------------|---------------|------------|
| TF-IDF (как сейчас)             | 1–3 сек     | 1–3 мин       | ~3 мин     |
| TF-IDF + code/fuzzy             | +1 сек      | 1–3 мин       | ~3 мин     |
| Эмбеддинги локально (CPU)       | 3–8 мин     | 30–90 сек     | ~5–10 мин  |
| LLM rerank топ-10 для всех 3000 | —           | 30–120 мин    | дорого     |
| LLM только для ~20% сомнительных| —           | 5–15 мин      | компромисс |

Вывод: 5000+3000 — малый/средний масштаб. Веб-приложение + Postgres/pgvector
справятся без тяжёлой инфраструктуры.

5. ТИПЫ ПРОБЛЕМ СЕМАНТИКИ И ИНСТРУМЕНТЫ
================================================================================

| Проблема                         | Пример                              | Инструмент              |
|----------------------------------|-------------------------------------|-------------------------|
| Разные формулировки              | кабель ВВГ vs провод силовой медный | Эмбеддинги + LLM rerank |
| Аббревиатуры без расшифровки     | ВВГ, СИП, ГОСТ                      | Словарь расширения      |
| Синонимы домена                  | труба ↔ изделие, кабель ↔ провод    | Словарь + эмбеддинги    |
| Тот же товар, другой код/имя     | код совпал, текст нет               | Code-first lookup       |

Один TF-IDF или один LLM — не оптимально. Нужен гибрид с настраиваемым режимом.

6. РЕКОМЕНДУЕМАЯ ЦЕЛЕВАЯ АРХИТЕКТУРА
================================================================================

Расширение через factory.py — routes.py и UI не переписываются.

                    ┌─────────────────┐
  item ────────────►│ CodeRetriever   │──┐
                    └─────────────────┘  │
                    ┌─────────────────┐  │    ┌──────────┐
                    │ TfidfRetriever  │──┼───►│  Merge   │
                    └─────────────────┘  │    │  (RRF)   │
                    ┌─────────────────┐  │    └────┬─────┘
                    │ EmbeddingRetriever│─┘         │
                    │ (pgvector)      │              ▼
                    └─────────────────┘   DeterministicFilter
                                                   │
                                         ┌─────────┴─────────┐
                                         │ HeuristicRanker     │ mode=fast/balanced
                                         │ LLMRanker (optional)│ mode=semantic
                                         └─────────────────────┘

6.1 Preprocessing (при импорте Excel)
  - нормализация + clean_code (с фиксом float)
  - опционально expand_abbreviations(text) — YAML словарь
  - опционально apply_synonyms(text) — добавление терминов
  - отдельное поле embedding_text

6.2 Три retriever'а + merge (Reciprocal Rank Fusion)
  - CodeRetriever: exact + fuzzy по коду (всегда)
  - TfidfRetriever: лексический recall
  - EmbeddingRetriever: семантика через pgvector

  Псевдокод RRF:
    scores[id] += 1 / (60 + rank)  для каждого списка кандидатов

6.3 Эмбеддинги
  Модели для русского/смешанного текста:
  - intfloat/multilingual-e5-small (баланс скорость/качество)
  - paraphrase-multilingual-MiniLM-L12-v2 (быстрее, слабее)

  SQL (pgvector):
    ALTER TABLE catalog_products ADD COLUMN embedding vector(384);
    CREATE INDEX ON catalog_products USING hnsw (embedding vector_cosine_ops);

  Индекс: один раз при загрузке каталога (~2–5 мин на 5000 строк CPU).
  Поиск: миллисекунды на запрос.

6.4 LLM reranker (опционально)
  Только для сомнительных позиций:
    if top1_score < 0.75 or (top1 - top2) < 0.05:
        rerank(top_10_candidates)

7. НАСТРАИВАЕМЫЕ ПАРАМЕТРЫ (ПО ЖЕЛАНИЮ)
================================================================================

Расширение RunMatchingRequest + UI:

  matching_mode: "fast" | "balanced" | "semantic"
    fast      = code + TF-IDF
    balanced  = code + TF-IDF + embeddings, без LLM
    semantic  = balanced + LLM rerank для сомнительных

  Опциональные параметры:
    top_k_candidates: int | None        (default 20, для semantic лучше 50)
    top_n_results: int | None
    min_similarity_score: float | None
    use_code_matching: bool = True
    use_tfidf: bool = True
    use_embeddings: bool | None           (None = из matching_mode)
    use_synonym_expansion: bool = False
    use_abbreviation_expansion: bool = False
    use_llm_rerank: bool | None
    embedding_model: str | None
    llm_rerank_threshold: float = 0.75
    synonym_dictionary: str | None      ("default", "electrical", ...)

Пресеты в UI:

  | Режим    | Что включено              | Время ~3000 | Семантика |
  |----------|---------------------------|-------------|-----------|
  | fast     | code + TF-IDF             | ~3 мин      | слабая    |
  | balanced | + embeddings              | ~8 мин      | хорошая   |
  | semantic | + LLM для ~20%            | ~15–25 мин  | лучшая    |

Словари: data/dictionaries/electrical_ru.yaml и др.

================================================================================
8. ОПТИМИЗАЦИИ ПОД МАСШТАБ
================================================================================

  1. Индекс один раз на запуск (TF-IDF cache, pgvector HNSW)
  2. Батч-эмбеддинги пачками 64–128 при импорте
  3. Семантика только в Postgres+pgvector (не SQLite)
  4. Параллельность: 4–8 воркеров для 3000 запросов (~3× ускорение)
  5. Меньший список перебирается, больший индексируется
  6. Инкрементальный re-embed только для изменённых строк
  7. RRF вместо max(score) — разные шкалы TF-IDF и embeddings

  Когда: разовые задачи, один человек, без ревью

12. БЫСТРЫЕ УЛУЧШЕНИЯ (v1.5) — ДЕТАЛИ

12.1 Code-first retriever (псевдокод)
  if item_code:
      exact = db.query(...).filter(code == item_code)
      fuzzy = rapidfuzz.extract(item_code, all_codes, score_cutoff=80)
      candidates += merge(exact → score 0.95, fuzzy → score 0.85)
  candidates += tfidf_top_k(20)
  candidates = deduplicate_by_product_id, keep max score

12.2 Фикс clean_code
  def clean_code(value):
      if isinstance(value, float) and value == int(value):
          value = int(value)
      text = str(value).strip().upper()
      ...

12.3 Fuzzy text retriever
  rapidfuzz.fuzz.token_set_ratio(item_name, catalog.name) по всему каталогу
  или по предфильтрованному подмножеству
  merge с TF-IDF через max(score) или RRF

12.4 Hybrid merge
  final = Reciprocal Rank Fusion([code_list, tfidf_list, fuzzy_list, embed_list])
