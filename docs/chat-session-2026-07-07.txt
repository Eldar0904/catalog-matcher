## Catalog Matcher — обзор проекта

**Catalog Matcher** — MVP-приложение для сопоставления внутреннего списка товаров с государственным или поставщическим каталогом. Работает без LLM: только TF-IDF, правила и fuzzy-сравнение кодов.

### Назначение

Типичный сценарий B2B/госзакупок:
1. Загрузить каталог (`Government_Catalog.xlsx`)
2. Загрузить свои позиции (`Our_Items.xlsx`)
3. Автоматически подобрать совпадения
4. Проверить и при необходимости скорректировать вручную
5. Экспортировать результат в Excel

### Стек

| Слой | Технологии |
|------|------------|
| Backend | Python, FastAPI, SQLAlchemy, pandas, scikit-learn, rapidfuzz |
| Frontend | React 18, Vite |
| БД | SQLite по умолчанию; PostgreSQL + pgvector в Docker |
| Инфра | Docker Compose (3 сервиса: `db`, `backend`, `frontend`) |

### Архитектура

### Движок сопоставления (конвейер)

Internal Item → TfidfRetriever (топ-20) → DeterministicFilter → HeuristicRanker (топ-3) → MatchResult в БД

Ключевая идея — заменяемые этапы через интерфейсы BaseRetriever, BaseFilter, BaseRanker. В v2 можно подключить эмбеддинги (pgvector) и LLM-ранжирование, меняя только factory.py.

### Модели данных

- CatalogSource — источник каталога
- CatalogProduct — строка каталога
- InternalItem — ваша позиция
- MatchResult — кандидат-совпадение
- MatchingRun — метаданные запуска

### API-эндпоинты

POST /api/upload/catalog, /api/upload/items, /api/match/run
GET /api/items, /api/catalog-products/search
POST /api/match/select, /api/export, /api/export/batches

### Frontend

Одностраничное React-приложение: загрузка файлов, топ-3 варианты, фильтры, manual override, цены в тенге (₸).

### Запуск

docker compose up --build
- Frontend: http://localhost:5173
- API docs: http://localhost:8000/docs

### Ограничения v1

- Нет семантических эмбеддингов и LLM
- Нет авторизации
- Тесты только для normalize.py

## Как работает сопоставление

Сопоставление — трёхэтапный конвейер для каждой позиции. Итог: топ-3 с оценкой уверенности и объяснением.

### 1. Подготовка данных

Каталог normalized_text: name + brand + model + description + technical_specs
Позиции normalized_text: item_name + description (код отдельно)

### 2. Запуск (POST /api/match/run)

Параметры: top_k=20, top_n=3, min_similarity_score=0.15

Pipeline: retriever.get_top_k → filter.filter → ranker.rank

### 3. TF-IDF Retriever

- TF-IDF матрица по каталогу, кэш на время запуска
- Cosine similarity, топ-20 с score > 0
- Биграммы ngram_range=(1, 2)
- Лексика, не семантика

### 4. Deterministic Filter

- Точный код → score min 0.95
- Похожий код (ratio > 80%) → +0.2
- Отсечение < 0.15

### 5. Heuristic Ranker

Сортировка, топ-3, лучший is_selected=1

### 6. UI уровни

≥60% зелёный, 30-59% жёлтый, <30% красный. «Подобрано» = ≥80%.

### 7. Ручная корректировка

Выбор из топ-3 или Manual override через поиск каталога.

### Сильные стороны

Точные названия, коды, русский/английский текст.

### Слабые стороны

Перефразировки, синонимы, семантически близкие но лексически разные описания.

## Алгоритм сходства — как устроен сейчас

### Шаг 0: что сравнивается

| Источник | Поля в normalized_text |
|----------|--------------------------|
| Каталог | name + brand + model + description + technical_specs |
| Позиции | только item_name + description |

Коды в текст не попадают — только на этапе фильтра.

### Шаг 1: TF-IDF + cosine similarity 0..1, топ-20

### Шаг 2: Filter — код boost, порог 0.15

### Шаг 3: Ranker — топ-3

## Почему пункты упускаются

1. Код проверяется только в топ-20 TF-IDF (архитектурная дыра)
2. Асимметрия полей (5 vs 2)
3. Нормализация "3x2.5" → "3x2 5"
4. top_k=20 — товар на 21+ месте теряется
5. БАГ: clean_code(12345.0) → "12345.0" vs "12345"
6. Нет fallback при пустом тексте с кодом
7. TF-IDF не ловит синонимы (ожидаемо для v1)

## Соответствие требованиям v1

| Требование | Статус |
|------------|--------|
| TF-IDF + правила | Да |
| Boost по коду | Частично (только в топ-20) |
| Точные названия | Да |
| Не пропускать по коду | Нет |

## Как улучшить

Быстро (v1.5):
1. Code-first retriever
2. Фикс clean_code для float
3. Код в normalized_text
4. top_k 50-100
5. Fuzzy по названию (token_set_ratio)
6. Симметрия полей

Средний: BM25, char n-grams, словари синонимов, hybrid merge

v2: EmbeddingRetriever (pgvector), LLMRanker

Приоритеты:
1. Code-first lookup
2. fix clean_code(float)
3. fuzzy text retriever
4. top_k + симметрия полей
5. embeddings (v2)

## Масштаб 5000 × 3000

Индексированный поиск, не 15 млн пар.

| Подход | Итого |
|--------|-------|
| TF-IDF | ~3 мин |
| TF-IDF + code/fuzzy | ~3 мин |
| Эмбеддинги CPU | ~5-10 мин |
| LLM все 3000 | дорого |
| LLM ~20% сомнительных | 5-15 мин |

## Инструменты по типу проблемы

- Разные формулировки → эмбеддинги + LLM rerank
- Аббревиатуры → словарь расширения
- Синонимы → словарь + эмбеддинги
- Код совпал, текст нет → code-first lookup

## Целевая архитектура

CodeRetriever + TfidfRetriever + EmbeddingRetriever → RRF Merge → Filter → Ranker/LLMRanker

Preprocessing: expand_abbreviations, apply_synonyms, embedding_text отдельно

Эмбеддинги: multilingual-e5-small, pgvector HNSW

LLM rerank только для сомнительных (top1 < 0.75 или малая разница top1-top2)

## Настраиваемые параметры

matching_mode: fast | balanced | semantic

Опции: use_code_matching, use_tfidf, use_embeddings, use_synonym_expansion,
use_abbreviation_expansion, use_llm_rerank, top_k, min_similarity_score, ...

Пресеты:
- fast: ~3 мин, слабая семантика
- balanced: ~8 мин, хорошая
- semantic: ~15-25 мин, лучшая

## Оптимизации

Индекс один раз, батч-эмбеддинги, Postgres не SQLite, параллельность 4-8 воркеров,
инкрементальный re-embed, RRF merge

## Варианты платформы

A. Веб-приложение (рекомендуется) — UI, pgvector, масштаб OK
B. Google Sheets + API — Sheets как витрина, не движок
C. Python скрипт + Excel — разовые задачи
D. Гибрид — FastAPI + React UI + опционально Sheets API

Сравнение: веб-app лучше для ревью топ-3 и семантики; Sheets для совместной работы;
скрипт для простых разовых задач.

Рекомендация: расширять текущее приложение.

## План этапов

Этап 1 (1-2 дня): code-first, clean_code fix, fuzzy, параметры
Этап 2 (3-5 дней): pgvector, EmbeddingRetriever, HybridRetriever, UI пресеты
Этап 3 (1-2 дня): YAML словари синонимов и аббревиатур
Этап 4 (2-3 дня): LLM rerank опционально

## Две таблицы 5000 и 3000

Меньший список перебирается, больший индексируется.
Сценарии: каталог+позиции, два поставщика, заявки к ТРУ.

## Итог

5000×3000 потянет. Семантика через factory.py + RunMatchingRequest.
Максимум: code-first + embeddings + словари + LLM выборочно.
